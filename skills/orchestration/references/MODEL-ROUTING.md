# Model and effort routing

Use current runtime metadata, official documentation, and local evals. Model names and availability drift; this file defines roles, not permanent vendor claims.

| Task class | Preferred role | Effort | Verification |
|---|---|---|---|
| Control plane / long program | strongest verified orchestrator model | highest available orchestration setting | state ledger, machine facts, independent advisors |
| Product, architecture, design, audit, deep planning | Fable 5 primary judgment advisor | `max` | blind memo + provenance; Tier 1 adds independent Sol Ultra |
| Adversarial strategy/challenge or Fable fallback | independent GPT-5.6 Sol | `ultra` | ephemeral/read-only, blind prompt, separate thread |
| Deep implementation, difficult debugging, native review | strongest verified Opus-class model | `high` normally; `xhigh/max` for high consequence | tests + cross-family review |
| Bounded implementation | strongest verified Sonnet-class model | `high` | exact scope, tests, current-head CI |
| Mechanical inventory/format/URL/fixture work | verified lightweight model | `medium`; raise on failure | deterministic command output |
| Security, money, tenant, credentials | two-family review plus user gate | maximum useful setting | machine invariants; no single model approves |
| Current research | strongest read-only research model | `high/xhigh`; multi-lane synthesis may use `ultra/max` | primary sources, dates, URLs, unknowns |

Raise model/effort when evidence conflicts, the task crosses several systems, consequences are irreversible, tests are unexplained, context loss is plausible, or reviewers disagree. Route by consequence, not diff size.

Do not use the largest model mechanically for deterministic work. Saving latency is allowed only when machine verification makes intelligence non-critical. Never downgrade a quality/security gate merely because the preferred model is unavailable.
